# Claude Agent Fleet — флот агентов

> 🇬🇧 English version: [`README.md`](README.md)

Это набор готовых **многоагентных проектов** для [Claude Code](https://claude.com/claude-code).
К каждому проекту приложена параллельная версия под **OpenAI Codex CLI** (см. [`CODEX.md`](CODEX.md)).

В репозитории лежит **только архитектура**: инструкции агентам, паттерны, шаблоны. Все рабочие
данные, выводы, память и личные/клиентские материалы вырезаны. Остался скелет, который вы
копируете, изучаете и адаптируете под себя.

---

## Что это и зачем — простыми словами

Представьте отдел. У отдела есть **руководитель** и несколько **специалистов**. Руководитель сам
не пишет код и не рисует презентации. Он берёт вашу задачу, разбивает её на части, раздаёт каждую
часть нужному специалисту, собирает результат и отчитывается перед вами. Специалисты не общаются с
вами напрямую и не лезут в чужую работу — каждый делает своё.

Каждый проект во флоте устроен ровно так:

- **Главный агент = руководитель (оркестратор).** Это файл `CLAUDE.md` в корне проекта. Он только
  раздаёт работу и говорит с вами.
- **Субагенты = специалисты (воркеры).** Это файлы в `.claude/agents/`. Каждый отвечает за одну
  роль: написать код, проверить код, найти источники, оформить PDF.

Зачем так, а не один агент на всё? Три причины:

1. **Чистый контекст.** Специалист по тестам не держит в голове всю переписку о дизайне. Он видит
   только свою задачу, поэтому работает точнее.
2. **Параллельность.** Руководитель запускает трёх специалистов сразу, если их задачи не зависят
   друг от друга.
3. **Контроль качества.** Один пишет, другой проверяет. Автор и ревьюер — разные агенты, и это
   ловит ошибки, которые автор не замечает.

## Какие проекты внутри

| Проект | Что делает | Специалисты (воркеры) |
|---|---|---|
| **`proj_arch`** | «Завод проектов» и хранитель архитектуры. Держит правила, по которым строятся все остальные проекты, создаёт новые проекты и их агентов, ведёт реестр флота. | `project-scaffolder`, `agent-author`, `pattern-curator`, `registry-keeper` |
| **`dev-team`** | Команда разработки полного цикла: разобраться в коде → спроектировать → написать → протестировать → отладить → отревьюить → проверить безопасность → задокументировать. | 9 специалистов (архитектор, разработчик, ревьюер, автор тестов и др.) |
| **`rnd`** | Исследовательский отдел: формулирует проверяемые гипотезы, запускает параллельных поисковиков, затем прогоняет синтез / «адвоката дьявола» / аудит. | 11 воркеров (планировщик, веб-исследователь, синтезатор, red-team-совет и др.) |
| **`mathematik`** | Маленький конвейер: фото заданий по математике → один свёрстанный PDF с полными решениями. | `vision-extractor`, `math-solver`, `pdf-composer` |
| **`smm-med`** | SMM-агент для медицинской клиники: контент-планы, посты, FAQ, работа с отзывами — в рамках закона о рекламе медуслуг. | `content-planner`, `content-writer`, `faq-builder`, `reviews-manager` |

Сердце флота — `proj_arch`. Его папка [`patterns/`](proj_arch/patterns/) — единственный источник
правды. Остальные четыре проекта собраны по этим правилам.

### Три правила, которые соблюдает каждый проект

1. **Главный агент только раздаёт работу** — всё реальное делают воркеры.
2. **Файлы агентов и навыков пишутся на английском**, а результат для пользователя — на языке
   проекта (например, посты `smm-med` на русском).
3. **Память проактивна** — каждый агент сначала вспоминает прошлый опыт, потом действует, и в
   конце записывает выводы.

## Установка на новую систему — пошагово

Раздел построен как инструкция, по которой пройдёт и человек, и сам агент. Ставьте только те
проекты, которые собираетесь запускать: каждый проект самодостаточен.

### Шаг 0. Что нужно поставить заранее

| Инструмент | Для чего | Установка |
|---|---|---|
| **Claude Code** | запускать версию под Claude | https://claude.com/claude-code |
| **OpenAI Codex CLI** | запускать версию под Codex | `npm i -g @openai/codex` ([docs](https://developers.openai.com/codex)) |
| **Node.js + `npx`** | MCP-серверы `sequential-thinking`, `context7` | https://nodejs.org (LTS) |
| **Python + [`uv`](https://docs.astral.sh/uv/) (`uvx`)** | MCP-серверы `fetch`, `headroom` | `curl -LsSf https://astral.sh/uv/install.sh \| sh` |
| **`git`** | склонировать репозиторий | уже стоит на macOS/Linux |

Большинство MCP-серверов **ставятся сами при первом запуске** через `npx`/`uvx` — заранее их
устанавливать не нужно. Руками ставится только то, что отмечено в таблице ниже.

### Шаг 1. Склонировать репозиторий

```sh
git clone https://github.com/renosaza/claude-agent-fleet.git
cd claude-agent-fleet
```

### Шаг 2А. Запуск под Claude Code

Claude Code сам находит `CLAUDE.md`, `.claude/agents/` и `.mcp.json` в той папке, где вы его
запустили. Поэтому:

```sh
cd dev-team          # или proj_arch / rnd / mathematik / smm-med
claude                # запустить Claude Code в этой папке
```

При первом запуске Claude Code спросит, разрешить ли MCP-серверы проекта — согласитесь на нужные.
`CLAUDE.md` загрузится как инструкция руководителя, а воркеры из `.claude/agents/` станут
доступны автоматически. Для проектов, где все серверы ставятся сами (`rnd`, `mathematik`), на этом
настройка закончена.

Для `proj_arch` (серверы `cloakbrowser` / `memorygraph`) и `dev-team` (сервер `codebase-memory`)
сначала загляните в раздел **MCP-серверы** ниже.

### Шаг 2Б. Запуск под Codex CLI

Codex читает инструкции из `AGENTS.md` (он уже лежит в корне каждого проекта), но кастомных
агентов держит **глобально**, в `~/.codex/agents/`, а не внутри репозитория. Из папки проекта:

```sh
cd dev-team

# поставить воркеров проекта как глобальных кастомных агентов Codex
mkdir -p ~/.codex/agents
cp codex/agents/*.toml ~/.codex/agents/

# поставить MCP-серверы и sandbox-конфиг как доверенный конфиг проекта
mkdir -p .codex && cp codex/config.toml .codex/config.toml

codex                # запустить; если Codex уже работал — перезапустите, чтобы агенты подхватились
```

Подробности, таблица соответствия моделей и предупреждение о коллизиях имён агентов — в
[`CODEX.md`](CODEX.md).

### Шаг 3. MCP-серверы — что чему нужно

MCP-сервер даёт агенту дополнительную способность: поиск в интернете, постоянную память,
управление браузером, разбор структуры кода. Вот что используют проекты:

| Сервер | Кто использует | Что делать при установке |
|---|---|---|
| `sequential-thinking` | все | ничего — `npx` ставит сам |
| `fetch` | все | ничего — `uvx` ставит сам |
| `context7` | proj_arch | ничего — `npx` ставит сам |
| `headroom` | proj_arch | ничего — `uvx` ставит сам (`headroom-ai[mcp]`) |
| `cloakbrowser` | proj_arch | **вручную** — укажите путь к своей установке или удалите блок |
| `memorygraph` | proj_arch | **вручную** — укажите путь к своей установке или удалите блок |
| `codebase-memory` | dev-team | **вручную** — поставьте бинарь одной командой (см. ниже) |

`cloakbrowser` и `memorygraph` используют пути-заглушки (`${HOME}/.claude/mcp-venvs/...`).
Поправьте их в `.mcp.json` (Claude) и `codex/config.toml` (Codex) под свою установку, либо удалите
блок сервера, если он вам не нужен. Проекты работают и без них — просто без этой способности
(например, без `memorygraph` не будет постоянной памяти между сессиями).

#### Сервер `codebase-memory` для dev-team

[`DeusData/codebase-memory-mcp`](https://github.com/DeusData/codebase-memory-mcp) — это сервер
«интеллекта по коду». Он индексирует кодовую базу в граф знаний (функции, классы, цепочки
вызовов, HTTP-роуты), чтобы агенты команды разработки отвечали на вопросы о структуре проекта без
перечитывания файлов. Память **раздельная по папкам**: каждая проиндексированная папка получает
свой граф (хранилище общее, в `~/.cache/codebase-memory-mcp/`). Запустили команду в одной папке —
там одна карта кода; переключились на другую папку — там своя.

Поставьте бинарь одной командой:

```sh
curl -fsSL https://raw.githubusercontent.com/DeusData/codebase-memory-mcp/main/install.sh \
  | bash -s -- --skip-config
```

Флаг `--skip-config` ставит **только** бинарь (в `~/.local/bin/codebase-memory-mcp`) и **не**
трогает глобальные конфиги ваших агентов. Сам сервер уже прописан в `dev-team/.mcp.json` и
`dev-team/codex/config.toml`, так что после установки бинаря он заработает.

После установки откройте проект и скажите агенту: **«проиндексируй этот проект»** (вызовется
инструмент `index_repository`). Граф построится для текущей папки. Дальше команда разработки
сможет искать по коду (`search_code`), смотреть архитектуру (`get_architecture`), отслеживать
цепочки вызовов (`trace_path`).

> Важно: сервер видит проект через явный вызов `index_repository` с путём к коду, а не
> автоматически по открытой папке. «Дать команде поработать с проектом в другой папке» = указать
> эту папку при индексации. Карты разных папок не смешиваются.

### Шаг 4. Проверить, что всё поднялось

- **Claude Code:** внутри проекта выполните `/agents` — должны быть видны воркеры; `/mcp` —
  должны быть подключены MCP-серверы.
- **Codex CLI:** выполните `/mcp` для списка серверов; сошлитесь на воркера по имени в запросе
  (например, *«пусть `code-reviewer` посмотрит этот диф»*), чтобы убедиться, что кастомные агенты
  загрузились.

## Как этим пользоваться — примеры

Со флотом вы общаетесь как с руководителем. Описываете задачу обычными словами, а он сам решает,
кого из специалистов привлечь. Имя воркера называть необязательно, но можно — тогда руководитель
отдаст работу именно ему.

### Пример 1. Написать фичу руками команды разработки

```sh
cd dev-team
claude
```

Вы пишете:

> Добавь в API эндпоинт `/users/{id}/avatar` для загрузки аватара. Валидируй размер и формат,
> покрой тестами.

Что произойдёт под капотом:

1. Руководитель попросит `explorer` разобраться, как устроен текущий код API.
2. `architect` предложит, куда вписать эндпоинт и как хранить файл.
3. `implementer` напишет код (и сам подчистит за собой по правилу «после фичи — рефакторинг»).
4. `test-author` напишет тесты.
5. `code-reviewer` прочитает диф и придерётся к слабым местам.
6. Руководитель соберёт всё и покажет вам результат.

Вы видите только итог и отчёт. Лазить по шагам не нужно.

### Пример 2. Заранее проиндексировать код, чтобы команда работала быстрее

> Проиндексируй этот проект, потом найди, где у нас собирается JWT-токен, и проверь обработку
> истёкших токенов.

Команда сначала построит граф кода (`codebase-memory`), затем ответит по структуре, не открывая
файлы один за другим. На большом репозитории это экономит много времени и контекста.

### Пример 3. Глубокое исследование

```sh
cd rnd
claude
```

> Разберись, стоит ли нам переходить с REST на gRPC для внутренних сервисов. Нужны аргументы за и
> против и оценка рисков.

Руководитель `rnd` сформулирует гипотезы, запустит параллельных веб-исследователей, затем прогонит
синтез и «адвоката дьявола», который попробует опровергнуть выводы. На выходе — отчёт с
источниками, а не просто мнение.

### Пример 4. Создать новый проект под свою задачу

```sh
cd proj_arch
claude
```

> Создай проект под генерацию еженедельных новостных дайджестов по заданным темам.

`proj_arch` — это завод. Он спроектирует декомпозицию (каких воркеров завести), создаст дерево
нового проекта по канону, напишет файлы агентов и обновит реестр флота. Готовый проект появится
рядом с остальными.

## Как устроены файлы проекта

```
<проект>/
├── CLAUDE.md            # инструкции руководителя (Claude Code)
├── AGENTS.md            # то же содержимое, для Codex CLI
├── .mcp.json            # MCP-серверы (Claude)
├── .claude/
│   ├── agents/*.md      # воркеры-специалисты (Claude)
│   └── skills/          # навыки (есть у dev-team)
└── codex/               # версия под Codex CLI
    ├── config.toml      # MCP-серверы + sandbox/approval
    └── agents/*.toml     # воркеры-специалисты (Codex)

proj_arch/
├── patterns/            # канон — одно правило на файл
└── templates/           # болванки, по которым завод собирает проекты
```

## Что вырезано из репозитория

Чтобы репозиторий было безопасно публиковать, из него убрали: все сгенерированные выводы и отчёты,
источники исследований, входные фото, реестр и журналы проектов, память агентов (memorygraph),
локальные файлы прав `settings.local.json`. Машинно-локальные пути заменены на заглушки: пара
MCP-серверов ссылается на `${HOME}/.claude/mcp-venvs/…`, и их нужно направить на ваши установки
(или удалить).
